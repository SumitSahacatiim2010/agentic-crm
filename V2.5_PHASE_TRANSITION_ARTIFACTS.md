**To**: Version 2.5 – Product Development

## Artifacts Requiring Update

### A. Product & Scope

| Artifact / Prompt | Why It Must Change | What Needs Updating |
|---|---|---|
| `Banking CRM Phase 2 Strategy Update.pdf` | Governs all phase execution — v2.0 scoped to demo validation, not production delivery | Add v2.5 product development objectives: multi-tenancy, customer-facing auth, real data ingestion pipelines, API rate limiting |
| `.cursorrules` (Project Governor) | References "Phase 2" demo constraints; lacks production engineering standards | Add production coding standards: error boundaries on all routes, structured logging, observability hooks, feature flags |
| `AGENTS.md` | Updated ✅ but still lacks v2.5-specific operational modes | Add: PRODUCTION MODE (performance profiling, load testing, canary deployments) |
| Product Roadmap (new) | Does not exist — v2.0 was phase-driven, v2.5 needs a sprint-based roadmap | Create: Sprint plan with milestones, feature toggle matrix, GA readiness checklist |
| Feature Scope Matrix (new) | v2.0 features were demo-grade — need explicit production/GA scope boundaries | Create: Feature-by-feature production readiness assessment (which features ship, which are deferred) |

### B. Architecture

| Artifact / Prompt | Why It Must Change | What Needs Updating |
|---|---|---|
| `next.config.ts` | Currently empty config — production needs CSP headers, image optimization, output config | Add: `output: 'standalone'` for containerized deployment, security headers, image domains |
| `.env` / `env.example` | Single environment (staging) — no dev/staging/prod separation | Create: `.env.development`, `.env.staging`, `.env.production` with separate InsForge project URLs |
| `Dockerfile` | Exists but untested for v2.5 production workload | Add: multi-stage build, non-root user, health check endpoint, proper NEXT_TELEMETRY_DISABLED |
| `deploy.sh` | Legacy GCP script — needs InsForge CI/CD alignment | Rewrite for InsForge `create-deployment` with environment-specific config |
| Scalability Doc (new) | No documented scalability assumptions — v2.0 assumed 200 parties | Create: Target scale (10K parties, 100 concurrent users, 500 journeys/day), DB indexing strategy |
| Middleware (`middleware.ts`) | Auth middleware exists but lacks rate limiting, CORS, request validation | Add: API rate limiting, request size limits, CORS configuration for production domains |

### C. Data

| Artifact / Prompt | Why It Must Change | What Needs Updating |
|---|---|---|
| `schema.sql` / `schema_leads.sql` / `schema_refinement.sql` | Fragmented across 3 files — no single source of truth for production schema | Consolidate: Single `migrations/` directory with numbered migration files |
| RLS Policies | v2.0 uses `anon` role for everything — no user-level access control | Add: User-level RLS policies keyed on `auth.uid()`, role-based access (RM, Compliance, Admin) |
| Seed Data Scripts | `scripts/generate_bulk_data.cjs`, `seed/`, `seed_data.sql` fragmented | Consolidate: Deterministic seed runner with environment flag (dev seed ≠ production) |
| `marketing_fatigue_log.channel` | No CHECK constraint — accepts arbitrary values | Add: `CHECK (channel IN ('email','sms','push'))` |
| Audit Logs schema | INSERT-only RLS but no retention policy or archival strategy | Add: Partition by month, retention policy documentation, export mechanism |
| DB Indexes (new) | No custom indexes beyond PKs — queries will degrade at scale | Create: Indexes on `leads.source_channel`, `marketing_fatigue_log.sent_at`, `compliance_profiles.aml_risk_rating`, `service_cases.status` |

## Recommended Order of Updates

| Priority | Artifact | Reason |
|----------|----------|--------|
| 1 | Environment separation (`.env.*`) | **Foundation** — all other work depends on knowing dev vs staging vs prod |
| 2 | Schema migration consolidation | **Foundation** — fragmented schemas are a deployment risk |
| 3 | `next.config.ts` (security headers, standalone) | **Security** — CSP, CORS required before any production exposure |
| 4 | RLS policy upgrade (user-level) | **Security** — `anon` role is demo-grade, not production-grade |
| 5 | Jest + Playwright configuration | **Quality gate** — automated tests must run before any code changes |
| 6 | DB indexes | **Performance** — queries will degrade at scale without indexes |
| 7 | Versioning strategy + CHANGELOG | **Process** — enables team collaboration and release tracking |
| 8 | CI/CD pipeline YAML | **Automation** — manual deployments are a production risk |
| 9 | Monitoring & alerting plan | **Observability** — can't operate production without visibility |
| 10 | Product roadmap + feature scope | **Alignment** — team needs to know what ships in v2.5 GA |
| 11 | Scalability documentation | **Planning** — informs infrastructure and architecture decisions |
| 12 | Backup & recovery | **Compliance** — regulatory requirement for financial data |
